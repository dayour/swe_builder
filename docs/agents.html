Agent Teams

You speak to the lead orchestrator. Behind the scenes, seven specialist sub-agents collaborate, debate, and produce validated artifacts. The lead delegates and executes. Specialists challenge each other before any change reaches Microsoft Copilot Studio.

Enabled via CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 in .claude/settings.json.

Operating rules

  • The lead does not generate instructions, YAML, or cards directly. Always delegates to teammates.
  • The lead does handle all MCS execution (Playwright, PAC CLI, Dataverse API). MCP access from teammates is unreliable.
  • QA Challenger reviews every teammate output before the lead executes it.
  • Teammates challenge each other. Bidirectional communication is the point.
  • All artifacts are persisted to files so the lead can read and execute them.

The seven teammates

Research

Research Analyst

Role: MCS capability researcher. Discovers what MCP servers, connectors, models, triggers, knowledge sources, and channels are available right now.

Why it matters: Microsoft ships features continuously. Static lists go stale fast. The Research Analyst searches MS Learn, the public web, community sources, and snapshots the live MCS UI before any architecture decision.

When invoked: /mcs-research Phase B (only if external systems need lookup), and on demand during /mcs-build when tool configuration hits issues.

Key strength: Prevents false limitation claims. If a feature exists in preview, the Research Analyst will find it.

Instructions

Prompt Engineer

Role: Instructions and prompt specialist. Primary role is writing MCS agent instructions. Secondary role is reviewing and sharpening skill files, agent definitions, and CLAUDE.md rules when they produce poor results.

When invoked: /mcs-research Phase C (writes instructions, single pass), /mcs-build on-demand (fixes during eval iteration), /mcs-fix (instruction-gap and boundary-violation fixes).

Key strength: Sharp, behavior-driven instructions with correct slash references to topics, tools, and child agents.

YAML

Topic Engineer

Role: MCS topic YAML and adaptive card specialist. Generates topic YAML for the code editor, designs adaptive cards, creates trigger configurations, and builds conversation flows. Deeply understands MCS YAML schema (357 types), all node types, PowerFx in cards, and channel-specific rendering limits.

When invoked: /mcs-research Phase D (validates topic feasibility), /mcs-build Step 4 (generates topic YAML), /mcs-fix (topic and trigger fixes).

Key strength: Syntax-correct YAML on the first try. Channel-safe adaptive cards. PowerFx that compiles.

Quality

QA Challenger

Role: Adversarial reviewer. The agent that asks "are you sure?" and "prove it works." Challenges every claim, tests against scenarios, validates cross-references, and finds gaps.

When invoked: After every teammate output (instructions, YAML, cards, architecture decisions). Also Phase D of research (generates eval sets) and Step 5.5 of build (build validation gate).

Key strength: Catches errors before they hit MCS. Classifies eval failures into root causes. Outputs verdict (PASS, PASS WITH CAVEATS, FAIL) that decides whether to ship or escalate.

Power Automate

Flow Designer

Role: Power Automate flow specification designer. Takes brief.json capabilities and designs complete flow specs with triggers, actions, connectors, data flow, and flow-manager.js commands. Writes specs only, never executes.

When invoked: When the brief includes Power Automate flows for agent integrations.

Key strength: Connector pattern fluency. Trigger and action composition. Flow JSON that imports cleanly via PAC CLI.

Sync

Repo Checker

Role: Repository sync validator. Scans all cross-references between app/, CLAUDE.md, README.md, skills, knowledge, settings, and package.json to find broken paths, stale docs, missing files, and drift.

When invoked: Tier 2 background after 3+ file changes or any code changes. Before any commit. After file reorgs, renames, or skill changes.

Key strength: Catches the cross-reference rot that compounds silently.

Audit

Repo Optimizer

Role: Repository optimization auditor. Finds dead files, duplicated code, oversized artifacts, stale scripts, unused exports, undocumented files, and unused dependencies. Reports findings only, never auto-fixes.

When invoked: On demand for cleanup passes. Before major releases.

Key strength: Conservative but thorough. Surfaces bloat without breaking anything.

Three-tier proactive quality model

Quality checks scale with risk. Not every response needs a full debate.

TierTriggerBlocks work?
Tier 1: Self-checkAny file editsNo, inline (10 seconds)
Tier 2: Repo Checker (background)3+ file changes or any code changesNo, runs async
Tier 3: QA ChallengeSchema change, workflow redesign, architecture decisionsYes, worth the 2-3 minute wait

Bidirectional review in practice

The lead spawns the team for a build. Teammates communicate directly:

QA → Prompt Engineer:
  "Instructions reference /OrderStatus topic that is not configured."

QA → Topic Engineer:
  "YAML node ID duplicated on line 14."

Topic Engineer → Prompt Engineer:
  "Your instructions expect Topic.orderStatus variable but no topic initializes it."

Research Analyst → Topic Engineer:
  "MCP server ContosoOrders just shipped. Use this instead of REST connector."

The lead reads validated artifacts from files and executes them via the hybrid stack.
