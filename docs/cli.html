CLI Reference

Every slash command, every flag. Slash commands are run inside the Claude Code terminal embedded in the dashboard or any standalone Claude Code session in the repo root.

Conventions

  • {projectId}: Folder name under Build-Guides/.
  • {agentId}: Folder name under Build-Guides/{projectId}/agents/.
  • Optional flags use --flag value format.
  • Commands are idempotent. Re-running picks up where a previous run left off (Phase 0 detection).

Workflow commands

/mcs-init

Create a new project folder structure and template files.

/mcs-init {ProjectName}

Output: Build-Guides/{ProjectName}/ with docs/, agents/, brief.json stub.

/mcs-context

Pull all M365 context for a customer via WorkIQ MCP. Searches emails, meetings, documents, Teams, people. Pre-fills 60 to 80 percent of the research phase.

/mcs-context {CustomerName}

Output: customer-context.md (narrative) plus customer-interactions.csv (timeline).

Prerequisite: WorkIQ CLI authenticated (workiq ask -q "test" for first-time setup).

/mcs-research

Read all project documents, identify agents, research MCS components, and produce fully enriched brief.json with eval sets. Smart at both project and agent level. Phase 0 detects new docs, brief edits, and manually created agents.

# Project-level (all agents)
/mcs-research {projectId}

# Agent-level (single agent)
/mcs-research {projectId} {agentId}

Reads: docs/, customer-context.md (if exists), knowledge/cache/, knowledge/learnings/.

Writes: brief.json (all fields including instructions and eval sets) plus evals.csv (derived flat export).

/mcs-build

Build and publish an agent in Copilot Studio using the hybrid stack. Handles find-or-create, step-level checkpoints for resume, MVP phase filtering, eval-driven iteration, and the QA validation gate.

/mcs-build {projectId} {agentId}

Reads: brief.json (the single source of truth).

Writes: brief.json.buildStatus (with completed steps for resume).

Routes by architecture: single agent runs standalone, multi-agent builds specialists first, then the orchestrator with child connections.

/mcs-eval

Run eval sets and write per-test results to brief.json. Three-tier execution: Direct Line API, Playwright Test Chat fallback, optional Native MCS Eval (async).

# All sets
/mcs-eval {projectId} {agentId}

# Specific sets
/mcs-eval {projectId} {agentId} --set critical,functional

# Native MCS eval (async, optional)
/mcs-eval {projectId} {agentId} --native

# Check Native eval results later
/mcs-eval {projectId} {agentId} --check-results

Output: brief.json evalSets[].tests[].lastResult plus evals-results.json.

/mcs-fix

Post-deployment fix. Analyze eval failures, classify root causes, apply targeted fixes (instructions, topics, evals), re-evaluate. For initial build iteration, use /mcs-build (which has an internal fix loop).

/mcs-fix {projectId} {agentId}

Five root-cause categories: instruction gap, boundary violation, routing failure, knowledge gap (manual), scoring issue.

Max 2 fix iterations per invocation.

/mcs-deploy

Deploy an agent from dev to test or prod. Two modes:

  • Agent-level (default, fast): uses replicate-agent.js to clone configuration directly.
  • Solution-level (ALM-ready): uses PAC CLI export and import for full ALM lifecycle.
/mcs-deploy {projectId} {agentId}

/mcs-report

Generate customer-ready reports from brief.json on demand without running a build.

# Brief report (design state)
/mcs-report {projectId} {agentId} --type brief

# Build report (status + deviations)
/mcs-report {projectId} {agentId} --type build

# Customer report (zero jargon)
/mcs-report {projectId} {agentId} --type customer

# Deployment report (checklists + env mapping)
/mcs-report {projectId} {agentId} --type deployment

/mcs-retro

Post-session retrospective. Captures learnings, updates the knowledge layer, syncs upstream tracked repos.

/mcs-retro

/mcs-refresh

Refresh one or all knowledge cache files with live research from MS Learn, web search, and MCS UI snapshots.

# Refresh everything
/mcs-refresh

# Refresh one file
/mcs-refresh triggers

/mcs-library

Browse, analyze, search, and contribute to the team SharePoint solution library. Uses SWE-Swarms for intelligent tagging and pattern extraction.

# Browse
/mcs-library list

# Search
/mcs-library search "knowledge agent"

# Analyze a downloaded solution
/mcs-library analyze {solutionId}

# Upload current project
/mcs-library upload {projectId}

Utility commands

/bug

File a bug report conversationally. Gathers details, previews, creates an ADO work item via az boards work-item create.

/bug

Auto-enriches with session context (project, agent, page, build status, eval score). Always previews before submitting.

/suggest

File a feature suggestion conversationally. Same flow as /bug but creates a Feature work item.

/suggest

PAC CLI quick reference

# List agents
pac copilot list

# Publish
pac copilot publish --bot <bot-id-or-schema-name>

# Status
pac copilot status --bot-id <bot-id>

# Extract template from existing agent
pac copilot extract-template --bot <bot-id> --templateFileName output.yaml

# Solution ALM
pac solution export --name "SolutionName" --path "Solution.zip"
pac solution import --path "Solution.zip" --publish-changes

# Auth
pac auth list
pac auth select --index 1

Tool CLIs

# YAML validation
.\tools\om-cli\om-cli.exe validate path\to\topic.yaml

# Constraint extraction
python tools/gen-constraints.py Microsoft.AdaptiveDialog

# Semantic gates
python tools/semantic-gates.py topics/OrderStatus.yaml --brief brief.json

# Drift detection
python tools/drift-detect.py brief.json

# Direct Line tests
node tools/direct-line-test.js --brief brief.json --set critical

# Eval scoring
node tools/eval-scoring.js --transcript run.json --rubric brief.json

# Multi-model review (optional)
node tools/multi-model-review.js --target instructions --file brief.json

# Solution library
node tools/solution-library.js list

# Upstream drift check
node tools/upstream-check.js

Environment

  • Working directory: Always run from the repo root.
  • Auth profiles: PAC CLI active profile must match the brief's environment (verified during build).
  • Browser session: Persisted profile reused by Playwright. Verified silently before every interaction.
