Schema-First, Identity-Bound

Schema is the contract. Identity is the perimeter. SWE_BUILDER is designed entirely around one canonical schema (brief.json) and one delegated identity token. Every canvas, every swarm, and every Microsoft Copilot Studio operation reads and writes through this single coordinated surface. There are no out-of-band channels. There are no stored service principal secrets.

NOTE: This page documents how the system holds together. If you remember one thing: the schema is the only contract that swarms and canvases share, and the auth token is the only credential that crosses canvas boundaries. Both are designed to be shareable as open-source patterns.

The contract: brief.json

brief.json is the single source of truth for an agent. It contains the intent, the design, the integrations, the eval suite, and the build state. Everything flows from it. Nothing operates without it.

  • Source-of-truth invariant: if it is not in brief.json, it does not exist for the build pipeline.
  • Validation invariant: every read and every write passes through JSON Schema validation. Garbage cannot enter the system.
  • Versioning invariant: schema is semver. Backward-compatible evolution only. Breaking changes ship a migration script.
  • Open invariant: schema lives at templates/brief.json in the repo. Public, forkable, MIT licensed.

Four canvases, one schema

Every UI surface in SWE_BUILDER is a canvas. Each canvas is a different view onto the same canonical schema. They never copy data, they never serialize across the network in alternate formats, and they never bypass validation. They render slices of brief.json and write slices back.

The token never leaves the user's session. It is delegated, scoped to MCS and Dataverse only, and short-lived. Every canvas refresh forces a token refresh through MSAL. There is no shared service principal, no API key, and no secret stored on disk.

The single auth token

One delegated token authenticates the user across every canvas and every downstream execution layer. It carries person details with it, so swarms and tools always know who is acting and on whose behalf.

Secure-by-design boundary model

Every layer enforces its own validation and authorization. There is no trust inheritance from the canvas to MCS. The token must satisfy each layer's policy independently.

Zero stored secrets

  • No service principal client secrets in .env or config files
  • No API keys for downstream connectors (delegated token only)
  • No persisted refresh tokens in plain text (MSAL handles cache encryption)
  • Token cache is per-user, per-machine, scoped by audience
  • Secrets that must exist (Direct Line keys for testing, optional GitHub Copilot tokens) live in OS-level credential managers, never in the repo

Single-point revocation

Revoke the user's Entra session and every canvas, every swarm context, every CLI operation, and every downstream MCS call stops within the token's TTL window (typically 60 minutes, often less when conditional access policies trigger).

  • One revocation in Entra propagates everywhere
  • No service principal to rotate independently
  • No long-lived API keys to hunt down
  • Audit trail in buildStatus.actor tells you exactly what the user did and when

Schema evolution

The schema evolves with discipline. Backward-compatible additions are routine. Breaking changes ship a migration.

flowchart LR V1[brief.json v1.0.0] -->|add field, optional| V11[v1.1.0] V11 -->|add field, optional| V12[v1.2.0] V12 -->|deprecate field| V13[v1.3.0] V13 -->|remove deprecated field| V20[v2.0.0] V20 -->|migration script| MIG[scripts/migrate-1.x-to-2.0.js] MIG -->|backfill defaults| OUT[Migrated brief.json]
  • Additive (minor bump): new optional field, new enum value, new section
  • Deprecation (minor bump): mark field as deprecated, keep it readable for one major version
  • Breaking (major bump): remove deprecated field, change required field, change enum semantics. Always paired with a migration script.

Validation gates

JSON Schema validation runs at five places:

  1. When the brief editor canvas saves a section
  2. When a swarm writes its slice back
  3. When the orchestrator merges swarm outputs
  4. When /mcs-build reads the brief at the start of a build
  5. When the build report is generated (sanity check before customer hand-off)

Open-source shareable patterns

The schema and the token shape are both designed to be shareable as open-source patterns. Other teams can adopt SWE_BUILDER's approach without copying the entire codebase.

OSS pattern

Schema as contract

Fork templates/brief.json as a starting point. Extend with your own fields. Publish your variant as a community schema. Validate with any standard JSON Schema validator.

OSS pattern

Delegated token only

Use the same token shape. Carry person details. Never store service principal secrets. The pattern works for any delegated-permission downstream API, not just MCS.

OSS pattern

Canvas as schema slice

Build canvases that render schema slices. Generate TypeScript types from JSON Schema. The UI adapts when the schema changes; you never sync types by hand.

Library: shareable schema instances

The solution library is a community of brief.json instances, each one a complete agent design. Browse them in the Library canvas, fork what you need, publish back what you build. The library is where schema becomes social.

  • 30+ cached solution snapshots in knowledge/solutions/
  • Each entry: brief.json + topics/ + evals.csv
  • Searchable by industry, capability, integration pattern
  • Upload to team SharePoint via tools/solution-library.js

Templates: schema scaffolds

The templates directory holds the canonical schema and the starter scaffolds. New projects begin from a template. Templates are versioned alongside the schema. 

templates/
  brief.json                # Canonical schema scaffold (the contract)
  topics/                   # Reusable topic YAML scaffolds
  evals/                    # Eval set scaffolds (5 default sets)
  flows/                    # Power Automate flow scaffolds

When you fork a template, you get a valid brief.json that passes schema validation on day one. Everything you change must continue to pass.

Framework: the assembled stack

The framework is the assembled stack: schema + canvases + swarms + lead orchestrator + hybrid execution layer. It is what you get when you put all the pieces together. Each piece is independently swappable.

The throughline

Schema is the contract that swarms and canvases share. Identity is the perimeter that operations cross. Both are open. Both are simple. Both are auditable. The system holds together because they hold together.
