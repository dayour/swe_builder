Workflow

Six phases drive every project from raw requirements to a published agent backed by an eval suite. The brief.json file is the single source of truth that flows through every phase.

flowchart LR A[Init] --> B[Context] B --> C[Research] C --> D[Build] D --> E[Evaluate] E --> F{Pass?} F -->|Yes| G[Deploy] F -->|No| H[Fix] H --> E G --> I[Report] I --> J[Retro] J --> C

Init

Creates the project folder structure and a starter brief.json.

/mcs-init MyAgent

Output: Build-Guides/MyAgent/ with docs/, agents/, and a placeholder brief. If SDR documents already exist in the folder, the skill detects them and converts .docx to .md via pandoc.

Context (optional)

Pulls all M365 history about a customer via WorkIQ MCP. Searches emails, meetings, documents, Teams, and people. Pre-fills 60 to 80 percent of research.

/mcs-context Contoso

Output: customer-context.md (narrative summary) and customer-interactions.csv (timeline).

Research

The most important phase. Reads docs, identifies agents, researches MCS components, and produces a fully enriched brief with eval sets.

/mcs-research MyAgent
/mcs-research MyAgent agent-name   # agent-level rerun

Four phases:

  1. Document comprehension and agent identification

    Lead reads all docs, cross-references with customer context, identifies agents (single or multi), extracts data entities, generates open questions for the customer.

  2. Component research (targeted)

    Lead resolves stable categories (channels, triggers, knowledge) from cache. Research Analyst is spawned only for external systems needing live MCP or connector lookup.

  3. Architecture and instructions

    Lead scores architecture (single vs. multi-agent) on six factors. Prompt Engineer writes instructions. QA Challenger reviews once.

  4. Eval sets and topic classification

    QA Challenger populates the five eval sets (critical, functional, integration, conversational, regression). Topic Engineer validates feasibility.

Smart at both levels
Phase 0 detects new or changed documents, brief edits, and manually created agents in MCS. Routes to full, incremental, re-enrich, or full-agent processing automatically.

Build

Constructs the agent in Copilot Studio using the hybrid stack with eval-driven iteration.

/mcs-build MyAgent agent-name

Build flow:

  1. Account and environment gate

    Verifies persisted account/environment match. Prompts only on first build or mismatch.

  2. Find or create

    Reuses existing agent ID if set. Otherwise checks pac copilot list for matching display name before creating new.

  3. Apply config

    Instructions via Dataverse API. Tools and connectors via Playwright. Knowledge upload via Dataverse. Model selection via Playwright.

  4. Topics

    Topic Engineer generates YAML, validates through four layers, lead pastes into Code Editor.

  5. Eval-driven loop

    Critical gate must pass 100 percent. Per-capability iteration runs each capability test set (max 3 iterations). Regression suite runs end-to-end (max 2 rounds). PE and TE auto-fix failures.

  6. Publish

    PAC CLI publishes the agent.

  7. QA validation gate

    QA Challenger validates brief vs. actual, every cross-reference, and deviation impact. Verdict (PASS, PASS WITH CAVEATS, FAIL) decides whether to generate the build report or escalate.

MVP filtering
Build only includes items tagged phase: "mvp". Items tagged phase: "future" are listed in the build report as deferred scope.

Evaluate

Runs eval sets and writes per-test results back to the brief.

/mcs-eval MyAgent agent-name
/mcs-eval MyAgent agent-name --set critical,functional

Three-tier execution:

TierMethodWhen used
1Direct Line APIDefault. Fast. Token auto-refresh, retry with backoff.
2Playwright Test ChatFallback. Drives the Test Chat pane in MCS UI.
3Native MCS EvalAsync. Uploads CSV to Evaluation tab. Use --native.

Eval methods include semantic similarity, exact match, regex, contains, must-not-contain, latency threshold, intent classification, and entity extraction.

Fix

Post-deployment fix loop. For initial build iteration, the loop inside /mcs-build handles fixes automatically. /mcs-fix is for production edge cases and regressions.

/mcs-fix MyAgent agent-name

Five root cause categories:

  • Instruction gap — fixed by Prompt Engineer
  • Boundary violation — fixed by Prompt Engineer (stronger guardrails)
  • Routing failure — fixed by Topic Engineer (trigger phrases)
  • Knowledge gap — manual; surfaces missing knowledge sources
  • Scoring issue — eval method tuning, not an agent bug

Deploy

Promotes the agent from dev to a target environment.

/mcs-deploy MyAgent agent-name

Two modes:

  • Agent mode — fast clone via tools/replicate-agent.js. Good for dev to test.
  • Solution mode — full ALM via pac solution export and pac solution import. Good for test to prod.

Includes pre-deploy validation, connection mapping, and a post-deploy smoke test.

Report

Generates customer-shareable summaries.

/mcs-report MyAgent agent-name --type customer

Four report types:

TypeAudienceContents
briefInternalFull design state, every brief field
buildInternalBuild status, deviations from brief, QA verdict
customerExternalPlain language, no jargon, what we built and why
deploymentOpsChecklists, environment mapping, rollback plan

Retro

Closes the loop. Captures learnings, updates the knowledge index, and syncs upstream changes.

/mcs-retro MyAgent agent-name

Two-tier capture model:

  • Tier 1 (auto): routine confirmations bump confirmed counts silently.
  • Tier 2 (user confirms): new discoveries, contradictions, architecture insights presented for approval.
